# Ledger API Contract (`@posteria/ledger` v0.1)

This contract defines the public API surface, audit-stream record shape,
configuration semantics, and short-circuit behavior of the open-source
Ledger subset. It is canonical for the externally observable behavior of
`@posteria/ledger` v0.1.

## Package Identity

- **Distribution name:** `@posteria/ledger` (npm, published under the
  `posteria` npm organization)
- **Repository:** `posteria-ai/observer` (GitHub, under the `posteria-ai`
  GitHub organization; separate from Posteria's commercial codebase). A rename
  to `posteria-ai/ledger` is planned but not yet effective.
- **License:** MIT
- **Initial version:** `0.1.0`
- **Runtime targets:** Node.js (LTS line current as of v0.1 cut); browser /
  edge runtime support is out of scope for v0.1.

## Decision Function

Ledger's decision function is the identity function. For every well-formed
request intercepted at the boundary (see "Malformed-input error path" below for
the sole exception):

1. Ledger MUST record a VDC-shaped audit record to the configured local
   audit stream.
2. Ledger MUST return a decision equivalent to `allow` with no side
   effects on the request payload.
3. Ledger MUST NOT evaluate policy. It MUST NOT consult any constitution,
   mandate registry, or remote policy server.
4. Ledger MUST NOT block, refuse, redact, or modify the request payload
   or any downstream response.

A conforming implementation that adds any policy evaluation, any blocking
behavior, or any payload modification is NOT `@posteria/ledger`; it is a
fork and MUST be renamed.

### Malformed-input error path

The decision-function obligations above apply to well-formed requests. A
request is **malformed** when, at either of two specific levels — the top-level
action object, or the direct `vdc` envelope object — it carries a field whose
name matches neither a v0.1 documented field at that level nor an
`x-<orgslug>-*` third-party extension. This includes any reserved `posteria_*`
top-level field and any reserved `vdc.*` envelope field (see "Reserved VDC
envelope extension fields").

This malformed-field check applies **only** to those two levels. It MUST NOT
recurse into nested values: in particular, the contents of `vdc.claims` are
opaque to Ledger (see "VDC envelope shape"), and arbitrary caller keys inside
`claims` are valid and MUST be preserved, never treated as malformed.

On malformed input, a v0.1 Ledger MUST reject the call before producing a
record:

1. It MUST raise a runtime error (throw) to the caller.
2. It MUST NOT write any audit record for that call — not a partial record, not
   a record with the offending field stripped.
3. It MUST NOT return `allow`.

This rejection is the producer's enforcement of the reserved/unrecognized-field
obligation defined in "Extension Hooks → Producer role". It is **not** policy
evaluation: Ledger rejects structurally non-v0.1 input before any decision is
formed, and consults no constitution, mandate registry, or policy surface to do
so. Rejecting malformed input is the only circumstance under which a v0.1
Ledger does not both return `allow` and produce a record; for every
well-formed request the identity-function obligations hold without exception.
Well-formed `x-<orgslug>-*` extensions are never malformed and MUST pass
through unchanged.

## Audit Stream Record Shape

Each record written to the audit stream MUST be a single JSON object with the
following fields. Unknown fields MUST be ignored by readers for decision-
making purposes. Extension semantics — including reserved Posteria namespaces,
third-party namespacing, and reader-side preservation obligations — are
specified in "Extension Hooks" below.

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `record_version` | string | yes | Strict SemVer (`MAJOR.MINOR.PATCH`). v0.1 emits `"0.1.0"`. |
| `record_id` | string | yes | Unique within the audit stream. UUIDv4 recommended. |
| `recorded_at` | string (RFC 3339) | yes | UTC timestamp of record write. |
| `action_kind` | string | yes | A short label categorizing the intercepted action (e.g. `"tool_call"`, `"http_request"`, `"mcp_invocation"`). |
| `action_signature` | string | yes | A stable, normalized representation of the action. Reader-facing identity; producers MUST NOT include user secrets in this field. |
| `vdc` | object | yes | Verifiable Digital Credential envelope. v0.1 contents are advisory and not yet attested; see "VDC envelope shape" below. |
| `decision` | string | yes | MUST be `"allow"` in v0.1. Reserved values: `"deny"`, `"clarify"`, `"escalate"` (not emitted by Ledger; reserved to avoid future collision). |
| `decision_reason` | string | yes | MUST be `"observer_short_circuit"` in v0.1. |
| `observer_version` | string | yes | The `@posteria/ledger` package version that wrote the record. |
| `host_metadata` | object | no | Optional process/host identifiers. MUST NOT contain user PII. |

### Extension Hooks

The audit record format reserves explicit namespaces and field positions for
future Posteria extensions and for third-party use. These reservations are
part of the v0.1 contract — implementations MUST honor them to remain
conforming, and readers MUST treat them as forward-compatibility hooks for
v0.2+.

**Reserved top-level fields.** The following names are reserved for v0.2+
Posteria-issued extensions. v0.1 Ledger MUST NOT emit any of them; a v0.1
record that contains any reserved field is non-conforming. v0.2+ MAY emit any
subset. Third parties MUST NOT use these names.

| Reserved field | Reserved purpose |
| --- | --- |
| `posteria_attestation` | Cryptographic attestation envelope over the record. |
| `posteria_signature` | Detached signature over the record body. |
| `posteria_signed_at` | RFC 3339 timestamp of signature production. |
| `posteria_policy_digest` | Reference to the closed-surface policy version that produced an associated decision, when paired with a non-Ledger Posteria component. |
| `posteria_linkage` | Cross-record linkage hash for tamper-evident chaining. |
| `posteria_extension_profiles` | Array of named extension-profile identifiers active on this record. |

**Reserved namespaces.**

| Prefix | Owner | Use |
| --- | --- | --- |
| `posteria_*` | Posteria | Reserved for Posteria-issued extensions, present and future. Closed to third parties. |
| `x-<orgslug>-*` | Third parties | Namespaced extensions by domain owner (e.g. `x-acmeco-trace_id`). The slug owner is responsible for collision avoidance within their own namespace. |

Extension-field obligations are split across three distinct roles. A single
tool MAY play more than one role provided it distinguishes the modes.

**Producer role (v0.1 Ledger).** A v0.1 Ledger MUST NOT emit any of the
six reserved `posteria_*` top-level fields or any field whose name matches
neither a v0.1 documented field nor an `x-<orgslug>-*` third-party extension.
A v0.1 Ledger that emits any such field is non-conforming. A v0.1 Ledger
enforces this obligation at the input boundary by rejecting any call that
supplies such a field, per "Decision Function → Malformed-input error path":
it throws and emits no record rather than producing a non-conforming one. The
acceptance suite at `test/conformance/` validates this obligation on the
producer side and MUST fail an implementation that emits a reserved or
unrecognized field.

**Reader role (data transport).** Readers consume audit records for
forwarding, transformation, storage, or display, without judging conformance.
Readers:

- MUST preserve all unknown and namespaced fields byte-for-byte (for strings)
  or by reference-preserving serialization (for objects/arrays).
- MUST NOT strip `posteria_*` or `x-<orgslug>-*` fields when forwarding.
- MUST NOT reject records or halt processing solely because a record contains
  a `posteria_*` field, an `x-<orgslug>-*` field, or an unrecognized field.
- MAY surface unknown fields to operators (logs, dashboards) for human
  review.

**Validator role (conformance checking).** Tools that validate v0.1
conformance of an Ledger implementation (the acceptance suite is the
canonical example) MUST enforce the producer obligation above and MUST flag
records that contain reserved `posteria_*` fields or unrecognized non-
namespaced fields as non-conforming when produced by a v0.1 implementation.
The validator role is distinct from the reader role; a tool acting as a
reader MUST NOT apply validator behavior to records it is merely transporting.

### VDC envelope shape

The `vdc` object in v0.1 carries advisory fields only. v0.1 does NOT
cryptographically attest, sign, or anchor VDC records. Fields:

| Field | Type | Caller obligation | Emitted record obligation | Notes |
| --- | --- | --- | --- | --- |
| `mandate_id` | string \| null | optional | always present | Caller-supplied mandate identifier; Ledger emits `null` if not supplied. |
| `issuer` | string \| null | optional | always present | Caller-supplied issuer identifier; Ledger emits `null` if not supplied. |
| `subject` | string \| null | optional | always present | Caller-supplied subject identifier; Ledger emits `null` if not supplied. |
| `claims` | object | optional | always present | Caller-supplied claim map. Opaque to Ledger. Ledger emits `{}` if not supplied. |

Ledger's transformation of the VDC envelope is **structural normalization
only**, not semantic interpretation:

- Ledger MUST emit the `vdc` object with all four fields present in every
  record, populating absent fields with the documented defaults above
  (`null` for `mandate_id` / `issuer` / `subject`, `{}` for `claims`).
- When a field is supplied by the caller, Ledger MUST emit the caller-
  supplied value byte-for-byte (for strings) or by reference-preserving
  serialization (for the `claims` object) without alteration, augmentation,
  or removal of caller-supplied keys.
- Ledger MUST NOT mint, sign, or attest VDC records in v0.1.

A future v0.2 may add attestation; v0.1 conformance forbids it.

### Reserved VDC envelope extension fields

The following names within the `vdc` object are reserved for v0.2+ attestation
and MUST NOT be emitted by v0.1 Ledger:

| Reserved field | Reserved purpose |
| --- | --- |
| `attestation` | Attestation block (issuer-signed) over the VDC content. |
| `signature` | Detached signature over the VDC body. |
| `signature_algorithm` | Identifier of the signing algorithm (e.g. `ed25519`, `es256`). |
| `attested_at` | RFC 3339 timestamp of attestation production. |
| `verifier_id` | Identifier of the party performing or validating attestation. |

A v0.2+ implementation that emits any of these MUST also produce a record
whose other `vdc` fields remain v0.1-conforming (i.e. attestation is additive,
never a replacement for the four advisory fields). Third parties MUST NOT use
these names within `vdc`; namespaced third-party extensions inside `vdc`
follow the same `x-<orgslug>-*` convention as the top-level record.

## Configuration Surface

A v0.1 conforming Ledger installation MUST accept configuration via these
mechanisms, in precedence order (later wins):

1. Programmatic constructor options (when used as a library).
2. Environment variables prefixed `POSTERIA_LEDGER_*`.
3. CLI flags (when used as a CLI).

Required configuration knobs:

| Knob | Type | Default | Notes |
| --- | --- | --- | --- |
| `audit_stream_path` | string (path) | `./posteria-ledger-audit.jsonl` | Local append-only file. Directory must exist; Ledger MUST NOT auto-create parent directories. |
| `enable_anon_telemetry` | boolean | `false` | When `true`, the no-op telemetry stub is engaged. v0.1 stub MUST remain no-op regardless of value; the flag exists to surface the future behavior to users. |
| `host_metadata` | object | `{}` | Optional per-installation labels. |

Unknown configuration keys MUST produce a startup warning but MUST NOT prevent
startup.

## Telemetry Stub Contract

v0.1 telemetry stub conformance requires:

- The stub is reachable only when `enable_anon_telemetry: true`.
- The stub MUST be a no-op in v0.1: it MUST NOT open network connections,
  MUST NOT write to remote endpoints, and MUST NOT exfiltrate any audit
  record content.
- The stub MUST NOT consume the contents of the audit stream.
- The package README MUST document, in user-facing voice, exactly what fields
  the future non-no-op telemetry would transmit if v0.2 enables it. v0.1
  installation transmits none of them.
- A future v0.2 enabling non-no-op telemetry requires a spec amendment and a
  major-version bump or explicit migration note. v0.1 MUST NOT be silently
  upgraded into a network-emitting build.

## Append-Only Semantics

The local audit stream MUST be append-only from Ledger's perspective:

- Ledger MUST NOT delete records.
- Ledger MUST NOT rewrite or rotate existing records in place.
- Ledger MAY support external log-rotation tooling by re-opening the file
  on SIGHUP or equivalent platform-conventional signal; the rotation itself
  is the host operator's responsibility, not Ledger's.
- Reader tools (separate from Ledger) MAY consume the stream destructively;
  Ledger makes no claim about reader-side semantics.

## Closed-Surface Non-Interoperability Statement

`@posteria/ledger` does NOT grant interoperability with, access to, or
licensed use of:

- The closed Posteria policy engine.
- The closed Posteria mandate issuance / VDC attestation surface.
- The closed Posteria constitution interpreter.
- The closed Posteria commercial server.

Audit records written by `@posteria/ledger` are intended to be consumable by
both Posteria-supplied readers and third-party readers under the MIT license.
Posteria reserves the right to evolve the closed-surface API and its
relationship to the Ledger record shape without notice. Ledger conformance
is defined by this contract, not by current or future closed-surface behavior.

## Conformance Test Expectations

A conforming v0.1 implementation MUST pass an acceptance suite covering:

- Identity-function decision behavior (every well-formed input → `allow` +
  recorded record; no payload mutation).
- Malformed-input rejection (see "Decision Function → Malformed-input error
  path"): an input carrying a reserved `posteria_*` top-level field, a reserved
  `vdc.*` envelope field, or any other non-namespaced field unrecognized at the
  top-level action or direct `vdc` envelope level → `record()` throws and emits
  no audit record. Well-formed `x-<orgslug>-*` extensions (at the top level and
  within `vdc`) and arbitrary `vdc.claims` contents are accepted and recorded.
- Audit-stream record shape validation against this contract.
- Telemetry-stub no-op verification (no network sockets opened, even with
  `enable_anon_telemetry: true`).
- Append-only semantics under SIGHUP / equivalent rotation events.
- Unknown configuration key → warning, not failure.

The acceptance suite ships in the OSS repository at `test/conformance/`.
