# v0.2 design: hash-chained audit stream via `posteria_linkage`

**Status:** proposal
**Tracking issue:** [#24](https://github.com/posteria-ai/ledger/issues/24)
**Target contract:** `docs/contract/v0.2.md` (new)
**Target record version:** `0.2.0`

## Summary

Populate the already-reserved `posteria_linkage` field with a per-record
chained hash so the audit stream becomes **tamper-evident under a
well-defined threat model**, without breaking Ledger's identity-function,
network-free, payload-non-mutating invariants. This is a contract-level
change targeted at `docs/contract/v0.2.md` (new) with a `record_version`
bump to `"0.2.0"`.

## Motivation

The v0.1 contract reserves `posteria_linkage` for exactly this purpose
(`docs/contract/v0.1.md:124`). v0.1 does not attest, sign, or anchor
(`README.md` "What Ledger is not"). The OSS competitive landscape has moved:
Microsoft's Agent Governance Toolkit (Apr 2026, MIT) and AgentBouncr (Feb
2026, hash-chained audit) now occupy the "verifiable audit trail" position.
To keep "tamper-evident" honest in Ledger's positioning without inflating
v0.1 claims, v0.2 should add chaining — and *only* chaining.

## Non-goals (explicitly excluded from v0.2 scope)

v0.2 MUST NOT add any of:

- Cryptographic signatures (`posteria_signature`, `posteria_attestation`,
  `posteria_signed_at` remain reserved-but-unemitted).
- Remote anchoring, transparency-log submission, or any network I/O.
- Payload mutation. `record()` remains the identity function returning
  `allow`.
- Policy evaluation, mandate minting, or VDC attestation.
- A second on-disk artifact. The chain travels in-band on each NDJSON
  record.
- Cross-file chaining. Per-file chain only (see "Rotation interaction"
  below).

These exclusions preserve the v0.1 "Scope Guardrails" from `AGENTS.md`.

## Threat model — stated precisely

v0.2 hash-chaining **detects** the following post-hoc modifications to an
existing audit stream file:

1. In-place edit of any byte of a recorded field.
2. Deletion of any record from the middle of the stream.
3. Insertion of a forged record into the middle of the stream.
4. Reordering of records.

v0.2 hash-chaining **does NOT defend against**:

- An attacker with write access to the *entire* file, who can recompute the
  chain end-to-end and re-serialize. Defense against this requires
  signatures or external anchoring, both explicitly out of scope.
- Truncation of the tail of the file. A reader sees a
  shorter-but-internally-consistent chain. Mitigation requires an external
  high-water mark or signed checkpoint, also out of scope.
- An attacker who controls the producer process at record time. Producer
  compromise is upstream of the chain.

This honesty is itself a differentiator. The README and contract MUST state
the bound. Ledger will not claim more than it provides.

## Specification

### Record shape change

A v0.2 record adds one required field:

```jsonc
{
  // ... all v0.1 fields ...
  "record_version": "0.2.0",
  "posteria_linkage": {
    "hash_algorithm": "sha-256",
    "prev_record_hash": "<hex|null>",
    "record_hash": "<hex>"
  }
}
```

- `hash_algorithm` — string, fixed to `"sha-256"` in v0.2. The field exists
  for future algorithm agility; readers MUST reject unknown algorithm
  values in v0.2.
- `prev_record_hash` — lowercase hex-encoded digest of the previous
  record's `record_hash`, or `null` for the genesis record.
- `record_hash` — lowercase hex-encoded digest over the canonical
  serialization of this record with the `posteria_linkage.record_hash`
  field stripped from the input.

`record_hash` covers `prev_record_hash` and `hash_algorithm`, so the chain
is integrity-bound at both ends of each link.

Hex encoding is chosen over base64url and multibase for readability and
uniformity. Encoding agility is not a goal of v0.2; algorithm agility lives
in `hash_algorithm`.

### Canonicalization

Hashing requires canonical serialization. v0.2 uses **RFC 8785 JSON
Canonicalization Scheme (JCS)** applied to the record object with
`posteria_linkage.record_hash` stripped from the input.

JCS is the only widely-implemented canonical JSON spec with a published
test suite. v0.2 takes a JCS implementation as a runtime dependency rather
than hand-rolling a subset. Canonicalization bugs are the load-bearing
failure mode of any chain — owning a custom canonicalizer would mean owning
that failure surface forever. The dependency cost is acceptable in
exchange.

### Genesis record

The first record in any audit stream file:

- MUST have `prev_record_hash: null`.
- MUST have `record_hash` computed normally.

This is the only sentinel value. The chain is per-file.

### Rotation interaction

`README.md` documents SIGHUP-driven log rotation (sink reopens fd). v0.2
uses **per-file chaining**: each rotated file starts a new genesis record
with `prev_record_hash: null`. The sink remains stateless across file
lifecycle, consistent with v0.1's "no internal consumer of its own audit
stream."

The blind spot is acknowledged: an attacker who can drop a whole rotated
file goes undetected by chain verification alone. Closing that gap requires
cross-file linkage or signed checkpoints, both deferred to a future
version. The v0.2 contract and README MUST state this limitation in the
threat-model section.

### Reader / validator obligations

Extends the v0.1 producer/reader/validator role split
(`docs/contract/v0.1.md:134-166`):

- **Producer (v0.2 Ledger):** MUST emit `posteria_linkage` on every
  record. MUST compute `record_hash` over JCS-canonicalized record with
  `record_hash` field omitted. MUST set `prev_record_hash` to the prior
  record's `record_hash`, or `null` for the genesis record.
- **Reader:** unchanged from v0.1 — readers transport records, do not
  validate the chain. A reader MAY surface chain field values; MUST NOT
  reject records based on chain state.
- **Validator (v0.2 chain verifier):** a new subrole, shipped as the
  `posteria-ledger verify <path>` CLI subcommand in the same
  `@posteria/ledger` package. Verification is part of the v0.2 contract
  surface; splitting it into a separate package would weaken the
  conformance story. The CLI MUST:
  - Walk the file, recomputing each `record_hash` and checking
    `prev_record_hash` linkage.
  - Report the position of the first chain break. Post-hoc the validator
    cannot distinguish edit / delete / insert / reorder; the report says
    "chain broken at record N."
  - Exit 0 on intact chain, non-zero on any break.
  - Be the canonical v0.2 conformance check alongside the existing
    `test/conformance/` suite.

### Reserved-field guard migration

`docs/contract/v0.1.md:113-116` and the runtime guard currently reject
records containing any reserved `posteria_*` field. v0.2:

- Lifts the guard for `posteria_linkage` only. The other five reserved
  `posteria_*` fields remain rejected on input.
- Caller-supplied `posteria_linkage` in `record()` input is still rejected.
  The chain is sink-managed, not caller-managed. The guard relaxation
  applies to the sink's own writes, not to the caller surface.

### Backward / forward compatibility

- `record_version` bumps to `"0.2.0"`. v0.1-only readers that strictly
  version-gate MUST reject v0.2 records (per
  `docs/contract/v0.1.md:100-102`).
- Readers that follow the v0.1 reader obligations (preserve unknown fields
  byte-for-byte) pass v0.2 records through correctly even without chain
  awareness.
- v0.2 producers MUST NOT emit v0.1 records and v0.2 records into the same
  stream file.
- Mixed-version stream files are non-conforming. The validator MUST flag
  them.

## Conformance test additions

New suite under `test/conformance/v0.2/`:

1. Genesis record has `prev_record_hash: null` and valid `record_hash`.
2. Each subsequent record's `prev_record_hash` equals prior record's
   `record_hash`.
3. Edit to any non-`record_hash` field of any record breaks the chain at
   that record.
4. Deletion of any non-final record breaks the chain at the following
   record.
5. Insertion of a forged record breaks the chain at that record.
6. Reordering two adjacent records breaks the chain.
7. JCS canonicalization edge cases: unicode normalization, number
   representation, key ordering, nested objects, arrays.
8. Unknown `hash_algorithm` value is rejected by the validator.
9. v0.1 record present in a v0.2 stream is flagged non-conforming.
10. Caller-supplied `posteria_linkage` on `record()` input is rejected at
    the boundary.

## Out of scope

- v0.3+ signature work (`posteria_signature`, `posteria_attestation`).
- External anchoring / transparency-log integration.
- Cross-file chaining.
- Real-time tamper alerting (Ledger is not an observability product).

## Deliverables

1. New `docs/contract/v0.2.md` containing the spec above.
2. README and LIMITATIONS updates stating the threat-model bound honestly.
3. JCS dependency adoption (selection of implementation pending; criterion:
   published test suite conformance).
4. Sink implementation of chain computation.
5. Reserved-field guard relaxation for `posteria_linkage` on sink writes
   only; caller-input rejection preserved.
6. `posteria-ledger verify` CLI subcommand in the same package.
7. `test/conformance/v0.2/` suite covering items 1-10 above.
